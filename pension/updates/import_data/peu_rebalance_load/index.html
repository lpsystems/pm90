Introduction

Rebalance switches are designed to allow the user to perform a mini lifestyle on the members investment portfolio for members whose investment set up is currently not lifestyling. This program allows the user to load data into Pension Master from an external data file onto the switch and investment history screens.

The program goes through a number of checking processes. Firstly the format of the data is checked. If any problems are found, e.g. a date value is trying to be put into a decimal field, then a formatting report will be generated informing the user of this. If no problems are found, then the program goes on to check that the data values are valid, e.g. if a record is being added which holds a Scheme Code, then the Scheme Code must be valid. If any problems are found, then an exceptions report will be generated informing the user of this.

The columns should be in the following order:

Column Field Conditions
a. Unique Identifier This column must have a label to indicate the identifier type. The column label should be on row one. If the unique identifier is Member Number, the label has to be Member Number, if it is Staff Number, the label has to be Staff Number and if it is PPS Number, the label has to be PPS Number. The column label is not case sensitive.
b. Surname The surname and forename values in column b and column c are not mandatory. But the columns must exist.If there are values in the columns, they will be checked to ensure they match the unique identifier.
c. Forename Linked to column b above.
d. Switch Date The switch date should not be earlier than the most recent rebalance switch for the member.
e. Expenses Values can only be Y or N.
f. Fund Codes The fund code columns must have a label on row 1 to indicate the fund code. The fund code labels must represent the fund codes on pension master and must be attached to the scheme. The fund code labels must be in upper case to reflect the fund codes on pension master. Up to 50 funds are catered for and should reside from column f on. If there is a 51st fund or more, you will be warned that the data will be will be ignored.
  • The file should only have the one header as detailed below. This header will facilitate the labelling of the columns as indicated in 2 above.
  • The file should have no footers.
  • The file should not have any blank lines.
  • The file should not be any larger than 32,000 rows.

If all the data lines held within the data file are correct, then the user can continue and load the data into the database. If not an exception report is created and the load it abandoned. Warnings will be issued in the following circumstances but the load will be allowed to proceed:

  • Member has positive un-posted transactions where units are null or zero.

  • Member has a lifestyling investment setup post the switch date.
